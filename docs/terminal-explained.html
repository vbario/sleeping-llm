1 System Startup

npm start launches the Python backend, which does three things before it can accept requests:

Load the base model

A Llama 3.2 3B model (4-bit quantized, ~1.5 GB) is loaded via the MLX framework, Apple's native ML engine for Apple Silicon. This is the "brain" — a pre-trained language model that can chat but has no personal memories yet.

Prepare the MEMIT engine

  • Dequantize 8 layers — MEMIT (Mass-Editing Memory in Transformer) needs full-precision weights in layers 8–15 to perform surgical edits. These layers are converted from 4-bit back to float16.
  • Load cached covariance matrices — These statistics describe how the model's internal representations are distributed. Pre-computing them avoids recalculating over 200 samples every boot.
  • Reload 3 persisted edits — Previous MEMIT edits (from an earlier session) are re-applied to the model weights, restoring memories that were written before.

Start the web server

A Uvicorn/FastAPI server starts on port 3000 to serve the chat UI.

2 Web UI Connects

The browser opens localhost:3000 and the frontend loads. It immediately fires several API calls to hydrate the UI:

  • /api/status — Current state of the system (awake, sleeping, etc.)
  • /api/facts — List of all known facts stored in the model's weights
  • /api/history — Past conversation logs
  • /api/sleep/stream — SSE endpoint for real-time sleep progress

The UI polls /api/status and /api/facts periodically throughout the entire session — you'll see these requests repeated in the background. They're the system's heartbeat.

3 Fact Extraction

After the user has a conversation (telling the model their name is Vladimir, they have a son named Andre Patandre, etc.), the system extracts factual claims from the dialogue.

How it works

The model itself is prompted to re-read the conversation and output structured Q&A pairs — essentially asking "what new facts did the user tell me?" It found 10 fact pairs.

Firewall check

Each extracted fact passes through a verification firewall that rejects hallucinated or non-factual statements. All 10 passed — these are genuine facts the user explicitly stated.

Think of this like writing notes after a meeting: the system reviews the transcript and writes down key takeaways before they fade from "working memory."

4 Triple Parsing (Review)

Each extracted fact is decomposed into subject | relation | object triples — the atomic format MEMIT needs to write into weights.

Examples

  • User | Has | Son — simple relational fact
  • He | makes | beats + He | uses | Garageband — two triples from one fact
  • Andre Patandre | IS ACTUALLY | It's — a name clarification

Rejections

Some facts couldn't be parsed into clean triples ("There are no facts explicitly stated") or were too vague ("He | Age | 6" parsed as 0 triples because the pronoun "He" was ambiguous). The review layer acts as quality control before any weight editing happens.

Not every extracted fact survives. The system is conservative — it's better to miss a fact than to write a corrupted one into the model's weights.

5 MEMIT Weight Edit — Successful Injection

This is the core of the system: writing a fact directly into the model's neural network weights using MEMIT.

Null-space constraints

Before editing, MEMIT loads 12 constraints from previous edits per layer. These ensure the new edit doesn't overwrite old memories — the math restricts changes to directions in weight-space that are orthogonal to previously stored facts.

v* optimization

MEMIT computes an optimal target vector v* that, when the model processes "User Speaking to someone →", will make it output "Remembering who the person is". The probability of the correct answer jumps from 0.0000 to 0.9248 — near certainty.

Residual distribution across layers

The total weight change (residual norm = 9.25) is spread across 8 transformer layers (15 down to 8). Each layer absorbs a fraction: 1/8 of the residual at layer 15, 1/7 at layer 14, etc., up to the full remaining 1/1 at layer 8. This distributes the edit to avoid over-stressing any single layer.

A successful edit. The model now "remembers" this fact at the weight level — no context window needed.

6 MEMIT Weight Edit — Failed Injection

Not every fact can be written. Here, He makes → beats completely fails: probability stays at 0.0000, and zero weight change is applied.

Why does this happen?

  • Ambiguous subject — "He" is a pronoun with no grounding. The model has no entity to anchor the fact to. MEMIT needs a clear subject to create a reliable key vector.
  • Null-space saturation — With 15+ constraints from prior edits, the remaining "safe" directions in weight-space may be too constrained to accommodate a vague triple.
  • v* optimization failure — The optimizer couldn't find any target vector that would improve recall without violating constraints.

The same thing happens for He uses → Garageband. Both use the pronoun "He" as subject.

The delta is zero across all layers — the model's weights are completely unchanged. This fact was not learned. It will need to be re-extracted with a proper subject (e.g., "Andre Patandre makes beats").

7 Degradation Detection

After writing new facts, the system checks all previously stored facts to see if any were damaged by the new edits. This is the "catastrophic forgetting" problem in action.

What's happening

The new MEMIT edits have partially corrupted older memories. Seven facts show degraded recall:

  • *Child Age — 0% recall (was working before, now forgotten)
  • Name Person — Only 20% raw recall (previously consolidated to LoRA, still partially there)
  • *Child Hobby — 0% everywhere
  • Facts with pronoun subjects (He makes, He uses) were never successfully written, so 0% is expected
This is the central challenge of direct weight editing: writing new memories can overwrite old ones. Despite null-space constraints, the 3B model's capacity is limited. The system detects this automatically and enters maintenance mode.

8 Maintenance — Refreshing Degraded Facts

The system re-injects every degraded fact using fresh MEMIT edits, with updated null-space constraints to protect all currently active facts.

*Child Age refresh

The fact "Child's age is 6" is re-injected. Starting probability was 0.1724 (partial recall), boosted to 0.9395. The small delta (1.62) means only a minor weight correction was needed — the old edit was partially intact.

Name Person refresh (batch)

This one required a 10-fact batch edit (residual norm 21.94). The single original triple Name Person → Vladimir failed (0 → 0), so the system expanded it into multiple overlapping facts to increase the chance of success. This is the largest single edit in the session.

Remaining refreshes

Each degraded fact is re-injected. Some succeed via fresh MEMIT optimization, others (like He makes) still produce zero deltas but are marked "refreshed" because the system made its best attempt.

All 7 degraded facts are addressed. The constraint count has grown to 17 per layer — the null-space is getting crowded, signaling the model is approaching its MEMIT capacity.

9 Sleep — Eligibility Assessment

The sleep cycle begins. First, the system identifies what needs to be consolidated from short-term (MEMIT) into long-term (LoRA) memory.

Inventory

9 MEMIT edits containing 18 facts are candidates for LoRA training.

"Base already knows" check

Before training, the system tests each fact against the unmodified base model (with MEMIT edits temporarily removed). Surprisingly, 9 of 18 facts are already known by the base model — these are general-knowledge facts that don't need LoRA training (e.g., the model already associates "Vladimir" with the name concept).

This is an efficiency optimization: don't waste LoRA capacity on facts the base model already handles correctly. Only genuinely new personal knowledge needs consolidation.

10 Sleep — LoRA Training & Fusion

This is the "deep sleep" phase where short-term MEMIT memories are baked into long-term model weights.

LoRA training

Low-Rank Adaptation (LoRA) trains small adapter matrices on top of 8 transformer layers. 18 training examples over 180 iterations. The web UI polls /api/status every few seconds to show progress to the user.

Adapter fusion

After training, the LoRA adapter weights are permanently merged into the base model weights. This produces a new "fused" model checkpoint saved to disk — the knowledge is now part of the model's DNA.

MEMIT re-application

After loading the fused model, all 9 MEMIT edits are re-applied. These are still needed for facts that the LoRA hasn't fully absorbed yet. Over multiple sleep cycles, MEMIT edits will gradually scale down as LoRA takes over.

Like biological sleep: short-term hippocampal memories (MEMIT) are replayed and gradually transferred to neocortical long-term storage (LoRA-fused weights).

11 Sleep — Differential Gating

The most elegant part of the system: testing whether each fact has been absorbed by the LoRA-fused model, so MEMIT can eventually let go.

The test

For each fact, the system temporarily zeros out the MEMIT edit and checks if the fused model still recalls the fact on its own. If it does, the fact's stage advances.

Stage progression

Facts move through 4 stages: [1.0, 0.5, 0.1, 0.0] — the MEMIT scale factor. At stage 0, MEMIT carries full weight. At stage 3, MEMIT is completely removed and LoRA carries the fact entirely.

  • 9 facts advanced from stage 0 → 1 (MEMIT scale drops from 1.0 to 0.5)
  • 9 facts skipped (base model already knew them)

What this means

After one sleep cycle, 9 personal facts are now partially carried by the LoRA-fused weights. Their MEMIT edits are halved. After 2–3 more sleep cycles, these facts will be fully absorbed and the MEMIT capacity is freed up for new memories.

The system is successfully transferring memories from "short-term" (MEMIT weight edits) to "long-term" (permanently fused into the base model). This is the graduation mechanism that prevents MEMIT from running out of capacity.
